## 🎨 프론트엔드 UI/UX 가이드 (DbManager)

DbManager 프로젝트에서 **새로운 화면/컴포넌트**를 추가할 때 따라야 할 UI/UX 가이드입니다.  
이 문서는 **프론트엔드 개발자**를 대상으로 하며, 실제 구현 시 `CONVENTIONS.md`, `USER_GUIDE.md`와 함께 참고합니다.

---

## 1. 전반적인 디자인 철학

- **일관성 우선**
  - 기존 레이아웃(`+layout.svelte`)과 홈(`+page.svelte`)의 스타일, 여백, 타이포그래피를 그대로 계승합니다.
  - Tailwind CSS 유틸리티 클래스를 사용하되, 비슷한 UI는 항상 **같은 클래스 조합**을 사용합니다.

- **데이터 관리 도구다운 심플함**
  - 정보 구조를 명확하게: **헤더 / 검색·필터 / 테이블 / 페이징 / 부가 패널(검증 등)** 순서를 유지합니다.
  - 장식보다 **가독성·검색성·정렬 가능성**을 우선합니다.

- **사용자 실수 방지**
  - 삭제, 덮어쓰기, 업로드 등 **파괴적 작업**은 항상 확인 모달 또는 확인 문구를 사용합니다.
  - 업로드, 매핑, 자동 생성 등은 항상 **결과 요약(성공/실패 건수)**와 **에러 메시지**를 보여줍니다.

---

## 2. 페이지 레이아웃 패턴

새로운 페이지는 다음 레이아웃 패턴을 기본으로 합니다.

- **상단 헤더**
  - 페이지 제목 + 한 줄 설명
  - 우측에 주요 CTA 버튼(예: “새 항목 추가”, “파일 업로드”) 배치

- **검색 & 필터 영역**
  - `SearchBar.svelte`, `ColumnFilter.svelte` 등 기존 컴포넌트 재사용을 최우선으로 합니다.
  - 검색창은 상단 헤더 바로 아래, **테이블 바로 위**에 위치합니다.

- **데이터 테이블 영역**
  - `DomainTable.svelte`, `TermTable.svelte`, `VocabularyTable.svelte`와 동일한 패턴:
    - 헤더 행 정렬 아이콘
    - 호버 시 행 하이라이트
    - 클릭/선택 시 행 강조

- **페이징 & 상태 영역**
  - 페이지 하단에 페이지네이션, 전체 건수, 현재 필터 상태(예: “필터 2개 적용 중”)를 배치합니다.
  - 로딩, 에러, 빈 상태(데이터 없음)를 명확하게 구분된 메시지로 표현합니다.

---

## 3. 네비게이션 및 라우팅

- **상단 글로벌 네비게이션**
  - `+layout.svelte`의 메뉴 구성과 스타일을 그대로 사용합니다.
  - 새 메뉴를 추가할 경우:
    - `menuItems` 배열에 `{ href, label, icon }` 객체를 추가합니다.
    - `icon`은 기존 `getIcon()` 함수에서 사용하는 SVG path 패턴을 재사용하거나, 유사한 단순 아이콘을 추가합니다.

- **현재 위치 표시**
  - `isCurrentPage(href)`처럼, 라우트와 `href`가 일치하면 **배경색과 텍스트 색을 변경**하여 활성 상태를 명확히 표시합니다.

- **URL 설계**
  - 리스트 페이지: `/entity/browse`
  - 업로드/관리 페이지: `/entity/upload`, `/entity/manage` 등 명확한 명사 기반 경로를 사용합니다.

---

## 4. 공통 컴포넌트 사용 가이드

- **검색바 (`SearchBar.svelte`)**
  - 가능한 한 **모든 리스트형 페이지**에서 동일 컴포넌트를 사용합니다.
  - Props/이벤트 네이밍은 기존 패턴(`query`, `field`, `onsearch`)을 그대로 따릅니다.

- **테이블 컴포넌트**
  - 신규 테이블은 `*Table.svelte` 패턴을 따라 만듭니다.
  - 공통 동작:
    - 열 정렬: 헤더 클릭 → 오름차순/내림차순 토글
    - 행 클릭: 상세 패널 열기 또는 편집 모달 열기
    - 체크박스: 일괄 삭제/다운로드 등 집합 연산에 사용

- **파일 업로드/관리 (`*FileManager.svelte`, `FileUpload.svelte`)**
  - 파일 선택 → 업로드 → 결과 요약 (성공/실패/스킵 건수) 순서를 유지합니다.
  - 업로드 결과는 항상 **표 형식 또는 리스트 형식**으로 제공하여 어느 항목이 실패했는지 명확하게 합니다.

---

## 5. 상호작용 & 상태 표현

- **로딩 상태**
  - 버튼/테이블/카드 등에서 비동기 동작이 있을 경우:
    - 버튼: “저장” → “저장 중…” 텍스트 + 비활성화
    - 테이블: 상단 또는 중앙에 `로딩 중...` 텍스트 또는 스피너

- **빈 상태(Empty State)**
  - 데이터가 없을 때는 단순히 “데이터 없음”이 아니라:
    - 짧은 설명 + 다음 액션 버튼(예: “첫 데이터 추가”)을 제공합니다.

- **에러 상태**
  - 에러 메시지는 가능한 한 **사용자 친화적인 한국어 문장**으로 표시합니다.
  - 기술적인 상세 메시지(스택 트레이스 등)는 콘솔 또는 로그에만 남기고 UI에는 노출하지 않습니다.

- **성공 피드백**
  - 저장/업데이트/삭제/업로드 성공 시:
    - 상단 또는 해당 영역에 **짧은 성공 토스트/배지**를 표시합니다.
    - “성공적으로 저장되었습니다.”, “3건이 삭제되었습니다.”처럼 **수량 기반 메시지**를 선호합니다.

---

## 6. UX 카피라이팅 가이드 (텍스트/라벨)

- **메뉴명 및 버튼 라벨**
  - 명사형/행동형을 명확히 구분:
    - 메뉴/섹션 제목: “단어집”, “도메인”, “용어”, “파일 관리”
    - 버튼: “저장”, “삭제”, “업로드”, “다운로드”, “동기화”, “자동 생성”

- **경고/확인 문구**
  - 삭제: “이 항목을 삭제하시겠습니까? 이 작업은 되돌릴 수 없습니다.”
  - 덮어쓰기: “기존 데이터를 덮어쓰시겠습니까? 기존 내용은 모두 사라집니다.”

- **에러 메시지 예시**
  - 검증 실패: “필수 항목이 누락되었습니다. 입력값을 확인해 주세요.”
  - 서버 오류: “일시적인 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.”

---

## 7. 반응형(Responsive) 및 접근성

- **반응형**
  - Tailwind의 반응형 프리픽스(sm/md/lg/…)를 사용해:
    - 모바일: 단일 컬럼, 카드형 UI
    - 데스크탑: 복수 컬럼, 테이블 중심 UI
  - 상단 네비게이션은 `+layout.svelte`처럼 **모바일 메뉴 버튼 + 드로어형 메뉴** 패턴을 재사용합니다.

- **접근성**
  - 버튼/링크에는 항상 `aria-label` 또는 스크린리더용 텍스트(`sr-only`)를 적절히 사용합니다.
  - 색상 대비는 Tailwind의 `text-gray-600` 이상, 배경 대비가 충분한 조합을 사용합니다.

---

## 8. 상태 관리 & 데이터 흐름 (UI 관점)

- **스토어 사용**
  - 파일 선택, 글로벌 설정 등 **다수 페이지에서 공유**하는 상태는 `src/lib/stores`에 Svelte store로 정의합니다.
  - 단일 페이지 내부에서만 사용하는 상태는 Svelte 5의 `$state` runes를 기반으로 **로컬 상태**로 처리합니다.

- **데이터 테이블 로직**
  - 정렬/필터/페이지네이션/검색은 `use-data-table.ts`의 패턴을 최대한 재사용합니다.
  - 새로운 데이터 테이블을 만들 때:
    - 공통 상태 구조 (`entries`, `loading`, `currentPage`, `pageSize`, `sortColumn`, `sortDirection`, `filters`)를 유지합니다.
